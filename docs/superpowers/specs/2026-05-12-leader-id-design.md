- **Issue**: #72 — feat: `@LeaderGroupElection` leaderId 지원 (LeaderGroupElector API 변경)
- **Date**: 2026-05-12
- **Branch**: `feat/leader-group-leaderid-72`
- **Worktree**: `.worktrees/feat-leader-group-leaderid`
- **Spec author**: Codex + feature-dev:code-architect (Opus)

## 1. 배경 (Problem)

`@LeaderGroupElection` AOP 어노테이션 (#41) 설계 시 `leaderId: String` 필드가 도입 후보였으나, Codex 리뷰에서 다음 지적:

> `LeaderGroupElection.runIfLeader(lockName, action)` API 는 leaderId 를 받는 자리가 없고, Redisson group 구현은 semaphore permit 만 획득. leaderId 가 어디에 전달되는지, 상태/소유권/해제 검증에 쓰이는지 정의 없음.

#41 본 PR 에서는 leaderId 필드를 제거했고, 본 이슈로 후속 작업. 본 spec 은 leaderId 의 **의미(semantic)**, **API 형태**, **백엔드 영향 범위** 를 확정하고, 8개 PR phase 로 분할 구현하는 청사진을 제공한다.

### 제약 / 미지(constraints)

- 7+ backend 모두 영향 (Local, Lettuce, Redisson, Mongo, Hazelcast, ZooKeeper, Exposed-JDBC, Exposed-R2DBC)
- 4개 execution model 모두 (sync / suspend / async / VT)
- AOP annotation 2개 (group + single) + Spring Boot autoconfig 1개
- 기존 `LeaderLease.leaderId` 의미(node identity) 와 신규 audit identity 충돌 가능

## 2. Design Risks / Failure Modes (8)

| # | Risk | 대응 |
|---|------|------|
| R1 | `LeaderLease.leaderId` 의미 변경(node id → election audit id) → 기존 consumer (Java serialization, JSON, BOM downstream apps, reflection) silent drift | **D3 revised** — `LeaderLease.leaderId` rename → `auditLeaderId`; legacy `leaderId` getter 가 `nodeId` 반환 `@Deprecated`; `serialVersionUID` bump `1L → 2L`; Phase A0 grep gate **확장** (worktree 내 + workshops/examples 내 downstream consumers); CHANGELOG release-note headline |
| R2 | Kotlin `runIfLeader(lockName, leaderId = "", action)` overload 시 JVM `$default` stub signature 변경 → Java 호출자 깨질 위험 | overload 회피, `LeaderSlot` value object 도입 |

| R7 | SpEL/property 가 blank/whitespace 결과 시 `LeaderSlot.init.requireNotBlank` throw → backend error 로 위장 | **D7 revised** — `resolveLeaderId` 각 level 에서 blank defensive check, fall through; level 4 provider blank → log + Default fallback; LeaderSlot 도달 전 non-blank 보장 |
| R8 | Level-4 autogenerated leaderId 가 명시적 설정값과 구별 불가 → multi-tenant audit forensics 불가 | **D10 신규** — `LeaderIdSource` enum (LITERAL/SPEL/PROPERTY/AUTO), `LeaderElectionInfo.leaderIdSource`, Micrometer tag `leader.id.source` 노출 |

## 3. 접근 비교 (3 approaches)

### Approach 1: Audit-only (CHOSEN)

- Backend ownership token (Redisson permitId / Lettuce Base58 / Mongo per-slot token) 그대로 유지
- `leaderId` 는 `LeaderLease.leaderId`, `LeaderRunResult.Elected.leaderId`, `LeaderElectionInfo.leaderId`, Micrometer metric tag, `LeaderLockHandle.Real.auditLeaderId` 등 **audit/관측** 표면에만 노출
- Caller 가 정의한 `leaderId` 는 락 소유권 검증에 사용되지 **않음** — 별도 metadata column/field 에 병행 저장

**Pros**: 4개 Redis 백엔드 변경 최소, Redisson 호환, 기존 ownership token 안정성 유지, multi-tenant audit 충분.
**Cons**: leaderId 가 release 검증 키가 아니므로 caller 가 "내가 holder 인지" 직접 확인 불가 (간접 — `LeaderLockHandle.auditLeaderId` 와 equal 비교).

### Approach 2: nodeId override per-call

- 기존 `LeaderGroupElectionOptions.nodeId` plumbing 재활용
- `runIfLeader` 호출 시 `nodeId` 일시 override

**Pros**: 신규 객체/필드 최소.
**Cons**: nodeId 의미 흐려짐 ("machine identity" vs "per-call audit id"), `LeaderLease.leaderId` 가 `nodeId` 와 어차피 동기화 — Approach 1 과 의미상 동일하지만 표면 더 혼란스러움. 명명·문서화 비용 큼.

**Rejected**: 의미 분리 명확성 부족.

### Approach 3: Replace ownership token

- Caller-supplied `leaderId` 가 release/extend 검증 키가 됨
- Local/Lettuce/Mongo 는 string equality 만으로 가능, **Redisson 은 `RPermitExpirableSemaphore` 포기 필요** → custom Lua semaphore 로 전면 재설계

**Pros**: leaderId 가 1급 시민, caller 직접 ownership 검증 가능.
**Cons**: Redisson 백엔드 fork 수준 redesign, 4개 Redis backend 의 일관성 깨짐, slot-token TTL 모델 전체 재검증. Risk R3 가 blocker.

**Rejected**: Redisson 백엔드 redesign 비용 > 직접 ownership 검증 가치.

## 6. 핵심 결정 (D1–D8)

### D1. `LeaderSlot` value object — 위치 / 형태

```kotlin
// leader-core/src/main/kotlin/io/bluetape4k/leader/LeaderSlot.kt
data class LeaderSlot(val lockName: String, val leaderId: String) : Serializable {
    init {
        lockName.requireNotBlank("lockName")
        leaderId.requireNotBlank("leaderId")
    }
    companion object {
        private const val serialVersionUID: Long = 1L
        fun of(lockName: String, provider: LeaderIdProvider): LeaderSlot =
            LeaderSlot(lockName, provider.nextLeaderId(lockName))
    }
}
```

- **`data class`** (not Kotlin `value class`): 백엔드/aspect/metric 에서 boxing 빈도 높음, inline boxing cost > 1 allocation

- 같은 패키지 (`io.bluetape4k.leader`) — `LeaderLease.kt` 옆 위치

### D2. `LeaderIdProvider` 인터페이스 / 구현

```kotlin
// leader-core/src/main/kotlin/io/bluetape4k/leader/identity/LeaderIdProvider.kt

/**
 * Strategy for generating per-election audit identifiers.
 *
 * ## Contract
 * - Implementations MUST NOT throw. Failure cases should be handled internally;
 *   if an implementation throws, the elector defensively falls back to
 *   [RandomLeaderIdProvider.Default] and logs the failure.
 * - Implementations MUST return a non-blank string. Blank results are rejected
 *   by the elector and the fallback chain advances.
 * - Implementations MUST be thread-safe.
 */
fun interface LeaderIdProvider {
    fun nextLeaderId(lockName: String): String
}

// RandomLeaderIdProvider — DEFAULT
class RandomLeaderIdProvider(private val length: Int = DefaultLength) : LeaderIdProvider {
    init { length.requireGt(0, "length") }       // NEW — fail-fast at construction
    override fun nextLeaderId(lockName: String): String = Base58.randomString(length)
    companion object {
        const val DefaultLength: Int = 12
        @JvmField val Default: LeaderIdProvider = RandomLeaderIdProvider()
    }
}

// HostnamePidLeaderIdProvider — opt-in (PII warning)
class HostnamePidLeaderIdProvider(private val suffixLength: Int = DefaultSuffixLength) : LeaderIdProvider {
    init { suffixLength.requireGt(0, "suffixLength") }       // NEW
    override fun nextLeaderId(lockName: String): String =
        "${LeaderNodeId.Default}:${Base58.randomString(suffixLength)}"
    companion object { const val DefaultSuffixLength: Int = 6 }
}

// CompositeLeaderIdProvider — prefix wrapping (tenant tag)
class CompositeLeaderIdProvider(
    private val prefix: String,
    private val delegate: LeaderIdProvider = RandomLeaderIdProvider.Default,
    private val separator: String = "-",
) : LeaderIdProvider {
    init { prefix.requireNotBlank("prefix") }
    override fun nextLeaderId(lockName: String): String =
        "$prefix$separator${delegate.nextLeaderId(lockName)}"
}
```

- Default bound bean = `RandomLeaderIdProvider.Default` (PII conservative)
- `HostnamePidLeaderIdProvider` KDoc: "Emits hostname:pid in audit logs; do NOT use in multi-tenant SaaS where hostname is sensitive."
- **Defensive elector wrapper** (R6 대응):
  ```kotlin
  protected fun safeNextLeaderId(lockName: String): String =
      try { idProvider.nextLeaderId(lockName).ifBlank { RandomLeaderIdProvider.Default.nextLeaderId(lockName) } }
      catch (e: Exception) {
          log.error(e) { "LeaderIdProvider failed; falling back to Default. provider=${idProvider::class.simpleName} lockName=$lockName" }
          RandomLeaderIdProvider.Default.nextLeaderId(lockName)
      }
  ```

### D3. `nodeId` vs `leaderId` — **rename strategy (revised)**

- `LeaderElectionOptions.nodeId` / `LeaderGroupElectionOptions.nodeId` = **node-level JVM identity** (default `host:pid`). **변경 없음.**
- `LeaderSlot.leaderId` = **per-election audit token** (default random Base58). 신규.

**`LeaderLease` schema 변경 (revised — R1 대응)**:

기존 `LeaderLease.leaderId` 필드는 backend 에 따라 nodeId 또는 fencing token 으로 사용 중이라 의미가 이미 흐려져 있음. Silent semantic shift 대신 **field rename + deprecated alias** 채택:

```kotlin
data class LeaderLease(
    val auditLeaderId: String,                        // RENAMED from leaderId (now per-election audit)
    val electedAt: Instant? = null,
    val leaseUntil: Instant? = null,
    val slot: Int? = null,
    val nodeId: String? = null,                       // NEW — additive (node-level JVM identity)
) : Serializable {
    init {
        auditLeaderId.requireNotBlank("auditLeaderId")
        nodeId?.requireNotBlank("nodeId")
    }

    /**
     * Legacy accessor for the previous `leaderId` field.
     * Returns the node identity ([nodeId]) when present, otherwise [auditLeaderId].
     * Existing callers reading `lease.leaderId` expecting `host:pid` should migrate
     * to [nodeId]; callers needing audit identity should use [auditLeaderId].
     */
    @Deprecated(
        message = "Use auditLeaderId for per-election audit identity, or nodeId for JVM node identity.",
        replaceWith = ReplaceWith("auditLeaderId"),
    )
    val leaderId: String get() = nodeId ?: auditLeaderId

    companion object {
        private const val serialVersionUID: Long = 2L   // BUMPED from 1L — semantic change
    }
}
```

- `serialVersionUID` 1L → 2L (R1 — serialization compatibility 명시)
- `@Deprecated val leaderId` 한 release cycle 유지 후 다음 major version 에서 제거

### D7. Annotation `leaderId` 해석 — fallback chain (revised)

**원칙 (Silent-failure blockers + Code-reviewer [high] 대응)**:
- Resolution failure 는 `LeaderIdResolutionException` 으로 항상 throw 됨 — `failureMode` 정책 **무관**. SpEL 코드 버그가 backend 장애로 silent misdiagnose 되는 패턴 방지.
- 각 level 에서 blank/whitespace 결과는 다음 level 로 fall through (R7 대응). Level 4 provider 가 throw 또는 blank → defensive `safeNextLeaderId` 가 `RandomLeaderIdProvider.Default` 로 fallback (R6 대응).
- Aspect 의 `resolveLeaderId` 호출은 **기존 try block 내부** 에 배치 — 단, `LeaderIdResolutionException` 만 별도 catch 하여 `failureMode` skip/fail-open 우회.

```kotlin
data class ResolvedLeaderId(val value: String, val source: LeaderIdSource)

enum class LeaderIdSource { LITERAL, SPEL, PROPERTY, AUTO }

class LeaderIdResolutionException(msg: String, cause: Throwable? = null) : RuntimeException(msg, cause)

private fun resolveLeaderId(
    meta: GroupAdviceMetadata,
    method: Method,
    args: Array<Any?>,
    target: Any,
    lockName: String,
): ResolvedLeaderId {
    // Step 0: Spring placeholder ${...} resolution mirrors how `name` is handled
    //         (existing precedent in resolveLockName / LeaderGroupElectionAspect.kt:514).
    //         placeholderResolver is the same StringValueResolver injected via SpEL evaluator.
    val literal = meta.leaderIdLiteral
        ?.let { spel.resolvePlaceholders(it) }
        ?.takeIf { it.isNotBlank() }
    if (literal != null) return ResolvedLeaderId(literal, LeaderIdSource.LITERAL)

    // Step 1: annotation SpEL — preParsed at startup by LeaderAnnotationValidatorBeanPostProcessor
    val spelExpr = meta.leaderIdExpression
    if (spelExpr.isNotBlank()) {
        val evaluated = try {
            spel.evaluate(spelExpr, method, args, target)
        } catch (e: Exception) {
            throw LeaderIdResolutionException(
                "SpEL evaluation failed for @LeaderGroupElection.leaderId='$spelExpr' on $method", e
            )
        }
        if (evaluated.isNotBlank()) return ResolvedLeaderId(evaluated, LeaderIdSource.SPEL)
        // blank result falls through (defensive — never throw on blank)
    }

    // Step 2: property fallback
    val prop = props.defaultLeaderId
    if (prop.isNotBlank()) return ResolvedLeaderId(prop, LeaderIdSource.PROPERTY)

    // Step 3: provider — defensively wrapped (safeNextLeaderId, see D2)
    val auto = safeNextLeaderId(lockName)
    return ResolvedLeaderId(auto, LeaderIdSource.AUTO)
}

// Aspect 의 sync/suspend/Mono 분기 — try block 내부 호출, dedicated catch
try {
    val lockName = resolveLockName(meta, method, args, target)
    val resolved = try {
        resolveLeaderId(meta, method, args, target, lockName)
    } catch (e: LeaderIdResolutionException) {
        // failureMode 무관하게 항상 throw — 코드 버그 silent misdiagnose 방지
        throw e
    }
    val slot = LeaderSlot(lockName, resolved.value)
    val info = LeaderElectionInfo(lockName, wasElected = true, leaderId = resolved.value)
    elector.runIfLeaderResult(slot) { /* action */ }
    // metric/info: resolved.source 로 LeaderIdSource 노출
} catch (e: CancellationException) {
    throw e
} catch (backendEx: Exception) {
    /* 기존 failureMode 정책 — backend 장애만 적용 */
}
```

- **Step 0**: Spring `${...}` placeholder 먼저 resolve — `name` 필드와 동일 precedent (Architect [medium] 대응)
- **Step 1**: SpEL eval. 실패 시 `LeaderIdResolutionException` throw (level 별 always-RETHROW)
- **Step 2**: property `bluetape4k.leader.aop.default-leader-id`
- **Step 3**: `LeaderIdProvider.nextLeaderId(lockName)` — `safeNextLeaderId` wrapper 사용 (custom provider throw / blank → `RandomLeaderIdProvider.Default` fallback + ERROR log)
- **`isNotBlank` 일관 적용** (Code-reviewer [medium]): whitespace-only 입력 fall-through
- **Idiomatic null+blank chain**: `?.let { spel.resolvePlaceholders(it) }?.takeIf { it.isNotBlank() }`
- **Provenance enum** (D10/R8): `ResolvedLeaderId.source` → `LeaderElectionInfo` 및 metric `leader.id.source` tag 노출

### D9. `LeaderLockHandle.Real.auditLeaderId` — 명시 결정 (Architect/Silent-failure [high])

```kotlin
sealed interface LeaderLockHandle {
    @InternalLeaderApi
    data class Real(
        val identity: LockIdentity,
        val token: String,                    // ownership token (release/extend key)
        val acquiredAtNanos: Long,
        val auditLeaderId: String? = null,    // NEW — nullable, "no audit" = null (not "")
        val slotId: String? = null,
        val reentryDepth: Int = 0,
        val extendDelegate: LeaseExtendDelegate? = null,
    ) : LeaderLockHandle {
        override fun equals(other: Any?): Boolean {                  // unchanged equality contract
            if (this === other) return true
            if (other !is Real) return false
            return identity == other.identity
                && token == other.token
                && reentryDepth == other.reentryDepth
                && slotId == other.slotId
            // auditLeaderId NOT included — metadata, not identity (Type-design [low] confirmed correct)
        }
        override fun hashCode(): Int = Objects.hash(identity, token, reentryDepth, slotId)

        fun withReentryDepth(depth: Int): Real = copy(
            reentryDepth = depth,
            // auditLeaderId is automatically preserved by data-class copy()
        )
    }
}
```

- **Nullable** (`String?` not `""`) — Silent-failure [high] / Type-design [high] consensus
- `equals/hashCode` 에서 제외 — value identity 는 `token` 이 결정 (audit 는 metadata)
- `withReentryDepth(depth)` 가 `data class copy()` 통해 `auditLeaderId` 자동 보존 — explicit test 필요 (Phase J)
- **모든 backend 의 `LeaderLockHandle.real(...)` factory 호출 site 업데이트 필요** — Phase 의 backend matrix (§8) 에 file 별 명시

## 13. Critical Details

### Error handling
- **Invalid SpEL** `leaderId`: startup 시 pre-parse 실패 → `IllegalStateException` (`name` 필드와 parity, strict 모드 무관)
- **Runtime SpEL eval failure**: `LeaderGroupElectionException` propagation → 기존 `failureMode` 정책 (RETHROW / SKIP / FAIL_OPEN_RUN)
- **Empty leaderId**: `LeaderIdProvider.Default` 가 항상 non-empty 보장 (Base58(12)); `LeaderSlot.init` 가 last-line of defense

### Threading
- `SpelExpressionEvaluator` 이미 Caffeine + thread-safe
- `RandomLeaderIdProvider` 내부 `Base58.randomString` → `ThreadLocalRandom` 기반, lock-free
- `LeaderLockHandle.withReentryDepth(n)` copy 생성자에 `auditLeaderId` 보존 필요 (`LeaderLockHandle.kt:91` 수정)

### Performance
- `LeaderSlot` 1 allocation/call: 기존 `LeaderLease` + `LeaderLockHandle.Real` + Base58 token 대비 negligible
- Literal `leaderId` SpEL short-circuit (`LITERAL_PATTERN`) — 기존 `name` 필드와 동일 fast-path
- SpEL eval 결과는 Caffeine cache per-method

### Security / PII
- **Default provider** = `RandomLeaderIdProvider` (no host/pid)
- `HostnamePidLeaderIdProvider` KDoc 명시 경고
- SpEL `leaderId` 평가는 `name` 과 동일 envelope: `bluetape4k.leader.aop.spel.allow-method-invocation=false` 기본

### Migration
- **`LeaderLease.leaderId` rename → `auditLeaderId`** (R1 / D3 revised) — `@Deprecated val leaderId` getter 한 release cycle 유지; `serialVersionUID` 1L → 2L; Phase A0 grep gate **확장** (worktree + workshops/examples + serialization consumer)
- `LeaderRunResult.Elected.leaderId: String?` nullable (D4 revised — Type-design [high] 대응)
- `LeaderElectionInfo.leaderId: String?` nullable + `wasElected=true` 시 non-blank 강제 (D5)
- `LeaderLockHandle.Real.auditLeaderId: String?` nullable (D9)
- `runIfLeader(lockName, action)` non-deprecated wrapper, `safeNextLeaderId` defensive 통과
- Exposed `audit_leader_id` 컬럼 — `leader-exposed-core` 공유 테이블 (group + single 모두), additive — 기존 row 영향 없음
- **Rollback test**: 새 코드가 audit 컬럼 쓴 row 를 옛 코드가 SELECT/INSERT/UPDATE 정상 처리하는지 Phase G 에서 확인 (Silent-failure [medium])
- **모든 leader-* 모듈 + downstream consumer 동시 recompile 필요** — `LeaderRunResult.Elected` data class default-arg 추가로 byte-code bridge 생성, 비호환 recompile 강제 (Architect [low] / CHANGELOG headline)
